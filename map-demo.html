Conversion

Now we need to convert the shapefile to a GeoJSON. Unfortunately I'm not sure this can be done within the JVM. However there is a command line tool called ogr2org that will do this automatically for you. It's part of GDAL

https://gdal.org/programs/ogr2ogr.html

The tool docs don't directly mention GeoJSON for some reason, but you can find conversion one liners on the GeoJSON page

https://gdal.org/drivers/vector/geojson.html

For instance, I went into the shoreline subdirectory GSHHS_shp/c and ran 

ogr2ogr -f GeoJSON shoreline.json GSHHS_i_L1.shp

This gives you a GeoJSON of the shorelines of all the continents (this seem to also include small periheral islands as well). I loaded both in QGIS and at least to my eye they look the same.

To display them ourselves we can create a processing "pipeline"

  • construct a "cropping polygon" which bounds our region of interest
  • use the intersect function in geo/jts to crop out the region of interest and get back a GeoJSON
  • flatten the GeoJSON/region's geometries into a bare list of polygons.

Sometimes intersect will cut a polygon into multiple polygons, and these will be grouped into a MultiPolygon object. You can also get GeometryCollection objects (unclear when). These too should be "unfolded". So that you are left with a long list of Polygons

There are some coordinate adjustments to rezero and flip the image 

(defn region-ranges
  [region]
  (let [{:keys [start-lat
                start-lon
                ended-lat
                ended-lon]} region]
    [(Math/abs (- ended-lat
                  start-lat))
     (Math/abs(- ended-lon
                 start-lon))]))

(defn crop-shoreline-to-region
  [shoreline
   region]
  (let [{:keys [start-lat
                start-lon
                ended-lat
                ended-lon]} region
        [lat-range 
         lon-range] (region-ranges region)
        area-of-interest (polygon-wkt [[start-lon start-lat
                                        start-lon ended-lat
                                        ended-lon ended-lat
                                        ended-lon start-lat
                                        start-lon start-lat]])]
    (->> shoreline
         (map (fn [shoreline-poly]
                (update shoreline-poly
                        :geometry
                        #(intersection %
                                       area-of-interest))))
         (filter #(not (.isEmpty (:geometry %)))))))

(defn geojson-to-svg
  [geojson
   region
   attribs]
  (let [{:keys [start-lat
                start-lon
                ended-lat
                ended-lon]} region
        [lat-range 
         lon-range] (region-ranges region)]
    (->> geojson
         (map :geometry)
         (reduce (fn [polygon-seq ;; extract polygons from multipolygons
                      next-geometry]
                   (println (.getGeometryType next-geometry))
                   (case (.getGeometryType next-geometry)
                     "Polygon"
                     (conj polygon-seq next-geometry)
                     "MultiPolygon"
                     (apply conj polygon-seq (polygons next-geometry))
                     "GeometryCollection" ;; This probably need to be recursive
                     (apply conj polygon-seq (geometries next-geometry))
                     ))
                 [])
         (map coordinates)
         (map coord-array) 
         (map (fn [poly]
                (map #(vector (- (.getX %)    ;; zero to starting longitude
                                 start-lon)
                              (- lat-range ;; SVG's Y axis is inverted
                                 (- (.getY %) ;; zero to starting latitude
                                    start-lat)))
                     poly)))
         (map #(svg/line-strip %
                               attribs))
         (apply svg/group {}))))

Rain Data

Now we want to read in rain data and overlay it. Rain data can be retrieved from arthurhou.pps.eosdis.nasa.gov and required an account.

The path on the server will look something like this:

ftp://arthurhou.pps.eosdis.nasa.gov/sm/730/gpmdata/2011/08/01/gis/3B-MO-GIS.MS.MRG.3IMERG.20110801-S000000-E235959.08.V06B.tif

And in depth explanation of the data is in a PDF here: https://gpm.nasa.gov/resources/documents/imerg-geotiff-worldfile-documentation

While the document focuses on the GeoTIFF format, it also provides all the necessary background to understand how everything fits together. It's a very good write up. Here I just provide a very short summary.

Looking at the file/path in parts:

  • sm/730/ unclear
  • gpmdata/ GPM is the current satellite system that synthesizes different data to give these precipitation maps
  • The gis in the path means GeoTIFF - b/c the original meteorology format is "HDF5" and it's not GIS software-friendly
  • The folders are organized by days. Each day's folder will have 48 half-hour snapshots of the precipitation. These come in a format that looks like 3B-HHR-…
  • Each day also has a cumulative GeoTiff, of all the precipitation for that day. This file has a slightly different name 3B-DAY-…
  • The monthly precipitation is hidden in the first day of the month's folder :). Its format will be 3B-MO-… All the other days of the month won't have this file.

The data can then be downloaded using CURL. The username and passwords provided by the service is very inconvenient b/c they contain a @ symbol, so you need to pass them in explicitely using the --user flag: 

curl -4 --ftp-ssl --user <<email>>:<<email>> ftp://arthurhouftps.pps.eosdis.nasa.gov/sm/730/gpmdata/2011/08/01/gis/3B-MO-GIS.MS.MRG.3IMERG.20110801-S000000-E235959.08.V06B.tif -o output.tif

We can do the same programmatically in Clojure 

;; Based off of libary example:
;; https://github.com/lsevero/clj-curl/blob/master/examples/simple_ftp.clj
(import (clj_curl.Handlers FileHandler))

(defn download-rain
  "Downloads precipitation GeoTIFFs from `arthurhou.pps.eosdis.nasa.gov`"
  [file-path
   file-name
   save-directory]
  (let [file-url (str "ftp://arthurhou.pps.eosdis.nasa.gov"
                      file-path
                      file-name)
        save-file (str save-directory
                       file-name)]
    ;;FileHandler needs to be closed after you finished to use it.
    ;;creating it with 'with-open' does that automatically for you.
    (with-open [filehldr (FileHandler. save-file)]
      (let [curl (curl-easy/init)]
        (do 
          (println "simple ftp")
          (curl-easy/setopt curl 
                            curl-opts/url
                            file-url)
          (curl-easy/setopt curl
                            curl-opts/writefunction
                            filehldr)
          (curl-easy/setopt curl
                            curl-opts/password
                            "geokon@qq.com")
          (curl-easy/setopt curl
                            curl-opts/username
                            "geokon@qq.com")
          (curl-easy/setopt curl
                            curl-opts/use-ssl
                            1) ;; magic value from curl header
          ;; https://github.com/curl/curl/blob/master/include/curl/curl.h#L875
          (curl-easy/perform curl)
          (println "speed download: "
                   (curl-easy/getinfo-double curl
                                             curl-opts/speed-download))
          (curl-easy/cleanup curl))))))

You will notice the file extension is .tif. Rain data arrived in two primary formats, HDF5 and GeoTIFF.

HDF5
is a more meteorologically specific format. This is how the data arrives orginally
GeoTiff
is an additional format provided. It's an image file along with some metadata

GeoTIFF

We can read in TIFF files directly in Java as long as it's Java 9 or newer. On older versions of Java you need to use a library/plugin that adds support.

The files provide precipitation data as a grayscale image of the whole globe. The higher the value of a pixel, the more mm of precipitation. In these particular files each pixel represents a 0.1 x 0.1 degree square. You have 1800 vertically and 3600 pixels horizontally - corresponding to the 180° of latitude and 360° of longitude.

The image is 16bit and single channel. This is a bit unusual and Java only supports these unsigned short (ie. 16bit positive numbers) values indirectly. If you start looking at a BufferedImage DataBuffer you will see what looks like an array of signed short values that range from -32767 to 32767. But we will want them as unsinged ranging from 0 to 65535. The easiest way is to use a helper function provided by Java, Short.toUnsignedInt, that will extract the value as if it were an unsigned short and copy it into an Int. We can then go through and copy out the whole buffer into a buffer of Ints before displaying them (we're only looking at a small region anyway)

Coordinate Chaos

  • A Java BufferedImage will naturally start with the [0,0] in the upper left-hand corner. This is the most common format for dealing with Images (and corresponds well to how you display things in a window - with the "root" always being the window's upper left-hand corner).
  • Geographic coordinates (such as in our GeoJSON files) come in [lat,long] pairs. The [0,0] point in the center (off the coast of Africa). Furthermore lat is the "height" on the map, while long is effectively the "width". This is the inverse of the usual ordering of [width,height].
  • Plots usually are drawn with [0,0] in the bottom left corner. (here I use a "heatmap" plot to display the rain "grid" data)

The code provides a pipeline to, crop out the region of interest, convert it to a matrix on integers, plot it as "heatmap" style svg image. Intensity is varied by changing each grid cell's luminosity

(defn geotiff-subregion
  "Extract a subregion from a GeoTIFF
  The GeoTIFF must cover the whole globe
  The REGION is described as a vector:
   [start-lat
    start-long
    ended-lat
    ended-lon]
  The result is a deep copy of the subregion in to a new `BufferedImage`"
  [geotiff
   [lat-res
    lon-res]
   region]
  (let [;; A lat-lon region is a square from bottom left to upper right corner
        {:keys [start-lat
                start-lon
                ended-lat
                ended-lon]} region 
        pix-per-deg-lat (/ 1.0
                           lat-res)
        pix-per-deg-lon (/ 1.0
                           lon-res)
        ;; An image region is top-left to bottom-right pixel
        start-pix-x (* (+ 180
                          start-lon)
                       pix-per-deg-lon)
        start-pix-y (* (- 90
                          ended-lat)
                       pix-per-deg-lat)
        width-pix (- (* (+ 180
                           ended-lon)
                        pix-per-deg-lon)
                     start-pix-x)
        height-pix (- (* (- 90
                            start-lat)
                         pix-per-deg-lat)
                      start-pix-y)]
    ;; This could be done without a deep copy..
    ;; But then you'd need to be more careful when getting the pixel at a later stage
    ;; b/c the databuffer will still be from the larger image ..

    ;; The deep copy is done according to this:
    ;; https://stackoverflow.com/a/36475402/7024671
    (-> geotiff
        (.getData (java.awt.Rectangle. start-pix-x
                                       start-pix-y
                                       width-pix
                                       height-pix))
        (.createWritableTranslatedChild 0 0)
        (#(java.awt.image.BufferedImage. (.getColorModel geotiff)
                                         %
                                         (.isAlphaPremultiplied geotiff)
                                         nil)))))

(defn geotiff-to-matrix
  "Take a GeoTIFF and convert it to a `thing/ndarray` matrix
  This is the format expected by the heatmap function"
  [geotiff]
  (let [int-buffer (->> geotiff
                        .getData
                        .getDataBuffer
                        .getData
                        (mapv #(Short/toUnsignedInt %)))
        max-value (apply max int-buffer)
        normalized-buffer (mapv #(* (/ 65535
                                       max-value)
                                    %)
                                int-buffer)]
    (->> normalized-buffer
         (#(ndarray/ndarray :int32
                            %
                            [(.getHeight geotiff)
                             (.getWidth  geotiff)])))))

(def grey-scale-for-shorts
  "A grayscale linear color map - over all possible `unsigned short` values
   see: https://colorcet.com/userguide/index.html"
  (->> (range 1
              0
              (- (/ 1
                    65535)))
       (mapv #(col/hsla 0
                        0
                        %
                        1))))

(defn matrix-to-heatmap
  "Draw a `heatmap' using a matrix of data.
  A width and height should be provided
  This way the SVG points stay in units of degrees
  This makes everything consistent (and it can be scaled later)"
  [matrix
   [lat-height
    lon-width]]
  (let [[height ;; which comes first was a bit unclear.. trial and error
         width] (ndarray/shape matrix)]
    {:x-axis (viz/linear-axis
              {:domain [0 width]
               :range [0 lon-width]
               :visible false})
     :y-axis (viz/linear-axis
              {:domain      [0 height]
               :range       [0 lat-height]
               :visible false})
     :data     [{:matrix        matrix
                 :value-domain  [0 65535]
                 :palette       grey-scale-for-shorts
                 :palette-scale viz/linear-scale
                 :layout        viz/svg-heatmap}]}))

(defn plot-rain
  "Returns a `thi.ng.geom` SVG hiccup (using a heatmap) of a GeoTIFF rain"
  [geotiff-file
   [lat-res
    lon-res]
   region]
  (-> geotiff-file
      java.io.File. 
      javax.imageio.ImageIO/read
      (geotiff-subregion [lat-res
                          lon-res]

(defn point-of-interest
  "Adds a scaled-down circle at a coordinate location.
  A region is needed to give appropriate scaling
  TODO : Make this work in the southern hemisphere"
  ([[lat
     lon]
    region]
   (point-of-interest [lat
                       lon]
                      region
                      nil))
  ([[lat
     lon]
    region
    attribs]
   (let [{:keys [start-lat
                 start-lon
                 ended-lat
                 ended-lon]} region
         [lat-range 
          lon-range] (region-ranges region)]
     (svg/circle [
                  (Math/abs (- lon
                               start-lon))
                  (Math/abs (- lat ;; need `abs` b/c of southern latitudes
                               ended-lat))]
                 (* 0.02
                    (min lat-range
                         lon-range))
                 attribs))))
